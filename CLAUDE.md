# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A marketplace of agent-agnostic skills. The canonical source for every skill lives in `skills/<name>/`. Everything else that contains skill content — the 12 harness directories, `plugins/`, `dist/` — is generated by the build and should never be edited directly. Edit `skills/<name>/`, run the build, commit the result.

Users install through `shdrs/skills` as a Claude Code marketplace, or clone/submodule the repo for other agents.

## Commands

```bash
node scripts/build.mjs     # build all skills for all 13 providers (always clean)
claude plugin validate .    # validate marketplace + plugin manifests
```

No dependencies to install. The build system uses only Node.js built-ins.

## Adding a new skill

1. Create `skills/<name>/SKILL.md` with YAML frontmatter (`name`, `description` required; `user-invocable`, `argument-hint`, `license`, `allowed-tools` optional).
2. Optionally add `skills/<name>/sub-skills/*.md` (loaded by the agent on demand) and `skills/<name>/applets/` (executable scripts copied verbatim).
3. Use `{{command_prefix}}`, `{{model}}`, `{{config_file}}`, `{{ask_instruction}}`, `{{applets_path}}` in `SKILL.md` body for values that differ per agent. These are resolved at build time.
4. Use provider-specific blocks (`<claude-code>...</claude-code>`, `<codex>...</codex>`, etc.) for agent-specific instructions. Non-matching blocks are stripped.
5. Run `node scripts/build.mjs`. This produces:
   - `dist/` intermediate output (gitignored)
   - 12 harness directories at repo root (`.claude/`, `.cursor/`, `.gemini/`, `.agents/`, `.github/`, `.kiro/`, `.opencode/`, `.pi/`, `.qoder/`, `.trae/`, `.trae-cn/`, `.rovodev/`) — committed
   - `plugins/<name>/` with `.claude-plugin/plugin.json` and Claude Code variant — committed
6. Add a plugin entry to `.claude-plugin/marketplace.json` with `"source": "./plugins/<name>"`.
7. Add `docs/<name>.md` with install instructions and usage.
8. Commit the canonical source, all harness dirs, the plugin subtree, and docs together.

## Architecture

### Three-layer structure

```
skills/<name>/          ← canonical source (human-authored)
  SKILL.md              ← frontmatter + body with template variables
  sub-skills/*.md       ← loaded on demand, not always in context
  applets/              ← executable scripts (Node.js ESM, .mjs)

scripts/build.mjs       ← reads skills/, writes dist/, syncs to harness dirs + plugins/
  lib/providers.mjs     ← 13 provider configs (configDir, providerTags, frontmatterFields)
  lib/utils.mjs         ← parseFrontmatter, compileProviderBlocks, replacePlaceholders

.claude-plugin/
  marketplace.json      ← marketplace manifest (users add via shdrs/skills)

plugins/<name>/         ← per-skill plugin subtree (built from Claude Code variant in dist/)
  .claude-plugin/plugin.json
  skills/<name>/...
```

### Build pipeline per provider

1. Parse YAML frontmatter, keep only fields the provider supports (defined in `providers.mjs` `frontmatterFields`)
2. Compile provider blocks in SKILL.md body (keep matching tags, strip others)
3. Replace `{{placeholder}}` tokens with provider-specific values from `PROVIDER_PLACEHOLDERS`
4. Replace `{{applets_path}}` with `<configDir>/skills/<name>/applets`
5. Copy sub-skills with provider block compilation only (no placeholder resolution — sub-skills may contain `{{...}}` as educational examples)
6. Copy applets verbatim (no transformation)
7. Sync from `dist/<provider>/<configDir>/skills/` to repo root harness dirs
8. Build plugin subtree from the `claude-code` variant

### Key design decisions

- **Sub-skills get provider blocks but NOT placeholder resolution.** This is because sub-skills may teach about template variables (showing `{{model}}` as literal examples). Only SKILL.md body gets full resolution.
- **Applets are copied verbatim.** No template resolution, no provider blocks. They run identically on every agent.
- **`.codex/` is not synced to repo root.** Codex reads from `.agents/skills/`, not `.codex/skills/`. The codex variant exists only in `dist/`.
- **Harness directories and `plugins/` are generated but committed.** They are build output, intentionally tracked so users can clone/submodule without running a build. The build replaces the entire `<configDir>/skills/` and `plugins/<name>/` directories on each run. Never edit these directly.
- **The `.github/` sync only touches `.github/skills/`.** Other `.github/` content (workflows, issue templates) is unaffected.

### Naming conventions

- `sub-skills/` not `reference/` — for markdown files loaded by the agent on demand
- `applets/` not `scripts/` — for executable code shipped with a skill (`scripts/` is reserved for the repo-level build system)
- `{{applets_path}}` not `{{scripts_path}}` — the template variable matching the directory name
